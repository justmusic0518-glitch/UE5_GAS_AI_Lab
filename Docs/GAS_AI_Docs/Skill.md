# GAS AI 진행 점검

## 범위

- 프로젝트 루트: `C:\Git Study\GAS_AI_Prototype`
- 문서 폴더: `C:\Git Study\GAS_AI_Prototype\Docs\GAS_AI_Docs`
- 목표 문서: `ProjectGoal.md`
- 일정 문서: `ProjectPlan.md`
- 구조 문서: `ProjectStructure.md`
- 기준 시간대: `Asia/Seoul`
- 자동화 실행 주기: 매일 오전 9시

## 점검 절차

1. `ProjectGoal.md`, `ProjectPlan.md`, `ProjectStructure.md`를 읽는다.
2. 문서 구조를 먼저 훑는다.
   - `Docs/GAS_AI_Docs` 안의 문서 목록을 확인한다.
   - 각 문서가 어떤 역할인지 짧게 파악한다.
   - 일정, 목표, 구조 문서가 서로 충돌하는 부분이 있으면 점검 결과에 표시한다.
3. 진행 상태를 판단하기 전에 프로젝트 구현 상태를 자세히 훑는다.
   - 텍스트/코드 파일 목록은 `rg --files`로 확인한다.
   - 우선 확인 대상은 `Source`, `Config`, `Content/_Project`, `Content/Map`, `.uproject`이다.
   - 생성물이나 소음이 많은 폴더는 제외한다: `.git`, `.idea`, `.vs`, `Binaries`, `DerivedDataCache`, `Intermediate`, `Saved`, `Content/__ExternalActors__`, `Content/__ExternalObjects__`.
   - 바이너리인 `.uasset` 파일은 직접 파싱하지 않는다. 파일 이름, 폴더 위치, 주변 설정으로만 추정한다.
4. `Asia/Seoul` 기준 오늘 날짜를 확인하고 `ProjectPlan.md`에서 해당 날짜의 계획을 찾는다.
5. 파일에서 확인 가능한 구체적인 증거를 기준으로 현재 어디까지 구현됐는지 판단한다.
   - 클래스 파일만 있고 동작이 거의 없으면 `부분 구현`으로 본다.
   - 다음 단계 진행을 막는 컴파일 위험, 플러그인/설정 누락, ASC Owner/Avatar 초기화 누락은 막힘 요소로 본다.
   - 완료된 항목은 내부 판단에는 사용하되, 출력에서는 기본적으로 생략한다.
   - 완료 항목은 일정 판단이나 다음 작업의 전제 조건으로 꼭 필요할 때만 한 줄로 언급한다.
6. 오늘까지 끝났어야 하는 계획과 현재 증거를 비교한다.
7. 일정 차이를 추정한다.
   - `정상`: 오늘까지의 핵심 항목이 확인되고, 다음 진행을 막는 위험이 보이지 않는다.
   - `미달 1일`: 하루치 핵심 산출물이 없거나, 동작하지 않는 뼈대만 있다.
   - `미달 2일 이상`: 주간 목표가 위험하거나, 여러 날짜의 산출물이 없거나, 다음 단계가 막혀 있다.
   - `빠름 1일 이상`: 미래 날짜의 산출물이 이미 구현되어 있고 실제로 사용할 수 있어 보인다.
8. `ProjectPlan.md`의 오늘 날짜와 현재 구현 상태를 기준으로 다음 작업 순서를 제시한다.
   - 목표는 작게 잡지 말고, 하루 안에 다 끝내면 다음으로 바로 넘어갈 수 있을 만큼 빡센 큰 목표로 잡는다.
   - `다음 진행`은 `## 1. 큰 목표명`, `## 2. 큰 목표명`처럼 번호가 붙은 강조 섹션으로 나눈다.
   - 각 큰 목표 아래에 `목표`, `완료 기준`, `권장 순서`, `추가 도전`을 적는다.
   - `목표`는 오늘 도달하고 싶은 넓은 기능 단위로 쓴다.
   - `완료 기준`은 에디터나 코드에서 확인 가능한 결과물 중심으로 쓴다.
   - `권장 순서`는 실제 작업 순서로 쓴다.
   - `추가 도전`은 사용자가 빨리 끝냈을 때 바로 이어서 할 수 있는 빡센 확장 목표로 둔다.
   - 일정이 밀렸다면 원래 계획을 그대로 밀어붙이지 말고, 큰 목표 안에서 오늘 반드시 살릴 핵심 성공 기준을 먼저 제시한다.
   - 일정이 빠르다면 다음 날짜 작업 중 바로 이어서 해도 되는 항목을 `추가 도전`으로 제시한다.
9. 사용자가 모를 수 있거나 헷갈릴 수 있는 개념이나 함수가 나오면 사용법을 설명한다.
   - Unreal/GAS/AI 관련 핵심 용어를 처음 쓰면 쉬운 설명을 먼저 붙인다.
   - 함수나 API는 “언제 쓰는지”, “어디에서 호출하는지”, “주의할 점”을 중심으로 설명한다.
   - 필요하면 현재 단계에서 바로 쓸 수 있는 최소 코드 흐름이나 호출 위치 예시를 붙인다.
   - 단순 용어 설명으로 부족하면 사용자가 혼자 공부할 때 볼 수 있는 문서 위치를 같이 연결한다.
   - `개념 메모`는 `## 1. 개념명`, `## 2. 개념명`처럼 번호가 붙은 강조 섹션으로 나눈다.
   - 각 개념 아래에 `쉽게 말하면`, `왜 지금 중요한지`, `헷갈리기 쉬운 점`, `지금 프로젝트에서 쓰는 방식`을 적는다.
   - 사용자가 대화하면서 더 물어볼 수 있도록, 너무 압축하지 말고 이해의 발판이 될 만큼 자세히 설명한다.
10. 오늘 작업과 직접 연결되는 학습 자료를 추천한다.
   - 추천은 `읽어볼 문서` 섹션에 넣는다.
   - 우선순위는 공식 문서, API Reference, 엔진 문서, 신뢰할 수 있는 문서형 자료 순서로 둔다.
   - Unreal Engine 버전 차이가 중요한 항목은 UE 5.5 또는 현재 프로젝트 버전과 맞는지 확인한다.
   - 가능하면 추천하기 전에 해당 문서를 직접 열어 내용을 확인한다.
   - 문서를 확인했다면 오늘 작업과 직접 관련된 부분만 요약하거나, 전체를 읽는 편이 좋으면 `전체 읽기 추천`이라고 표시한다.
   - 링크를 확인할 수 있으면 제목과 링크를 함께 적고, 어떤 부분을 읽으면 되는지 적는다.
   - 링크를 확인할 수 없거나 인터넷 접근이 제한되면 무리하게 URL을 만들지 말고 검색어와 확인 필요 표시를 제시한다.
   - YouTube 영상은 기본 추천하지 않는다. 사용자가 영상 자료를 명시적으로 요청했을 때만 추천한다.
   - 블로그나 오래된 글은 공식 문서가 부족할 때만 보조 자료로 추천하고, 버전이 오래됐을 수 있음을 표시한다.
11. 점검은 최고 수준으로 신중하게 한다.
   - 표면적인 파일 존재 여부만 보지 말고, 다음 단계 진행을 막는 구조적 위험이 있는지 본다.
   - 확실하지 않은 내용은 추정이라고 표시한다.
   - 가능한 한 충분히 자세히 출력한다. 출력 길이를 억지로 줄이지 않는다.

## 출력 규칙

- 첫 부분에는 아래 내용을 포함한다.
  - `정상`, `미달 1일`, `미달 2일 이상`, `빠름 1일` 같은 일정 상태
  - 파일에서 확인한 구체적인 근거와 추정 근거
  - 현재 단계에서 가장 먼저 확인할 것 하나
- 그 다음에는 `문서 구조 확인` 섹션을 넣고, 프로젝트 문서들이 무엇을 담당하는지와 충돌 여부를 적는다.
- 그 다음에는 `구현 상태 확인` 섹션을 넣고, `부분 완료`, `미완료`, `막힘`만 나눈다. `완료` 항목은 기본적으로 출력하지 않는다.
- 그 다음에는 `다음 진행` 섹션을 넣고, 지금 단계에서 밀어붙일 큰 목표를 `## 1. ...`, `## 2. ...` 형식의 번호별 강조 섹션으로 안내한다.
- `다음 진행`의 각 큰 목표에는 `목표`, `완료 기준`, `권장 순서`, `추가 도전`을 포함한다.
- 필요한 경우 `개념 메모` 섹션을 넣고, 오늘 작업에 직접 관련되거나 헷갈리기 쉬운 개념과 함수 사용법을 자세히 설명한다.
- `개념 메모`도 `## 1. ...`, `## 2. ...` 형식의 번호별 강조 섹션으로 나눈다.
- `개념 메모`의 각 항목에는 가능하면 `쉽게 말하면`, `왜 지금 중요한지`, `헷갈리기 쉬운 점`, `지금 프로젝트에서 쓰는 방식`을 포함한다.
- 필요한 경우 `읽어볼 문서` 섹션을 넣고, 오늘 막힐 가능성이 높은 개념에 대한 문서형 학습 자료를 추천한다.
- `읽어볼 문서`에는 가능하면 문서를 직접 확인한 뒤, 읽어야 할 부분과 이유를 적는다.
- 전체적으로 자세히 출력한다. 출력 길이를 이유로 중요한 점검 근거, 다음 단계, 개념 설명, 참고 문서를 생략하지 않는다.
- 사용자가 리뷰를 요청하지 않았으면 전체 코드 리뷰를 하지 않는다.
- 일일 점검 중에는 큰 리팩토링 제안을 하지 않는다.
- 사용자가 구현을 명시적으로 요청하지 않았으면 일일 점검 중 파일을 수정하지 않는다.
